3 Credit Card Fraud Detection

Fraud modeling is hard for a practical reason: the transactions you care about are rare, the losses can be large, and many legitimate transactions also look irregular in the raw data. Travel, holiday spending, or an unusually large purchase can resemble fraud unless the model has enough context to tell the difference.

3.1 Card-present frauds

  • Card-present (CP) fraud occurs when a transaction is made using a physical credit card that has been lost, stolen, counterfeited, or intercepted.
  • The legitimate cardholder is typically unaware of the fraudulent use until they notice unauthorized transactions on their statement or are alerted by their bank.
  • What makes CP fraud distinct is that the physical card is used at the point of sale (e.g., a retail store or restaurant).

3.2 Card-not-present frauds

  • Card-not-present (CNP) fraud occurs when credit card information (such as card number, expiry date, CVV) is used without the physical card being present.
  • This type of fraud is common in online purchases.
  • Fraudsters may obtain card details through data breaches, phishing scams, or malware.
  • CNP fraud has become increasingly prevalent with the growth of e-commerce.

3.3 Credit Card Fraud Detection Systems (CCFDS)

  • Credit Card Fraud Detection Systems (CCFDS) score transactions in real-time or near real-time using signals such as amount, merchant or terminal history, timing, location, and recent customer activity.
  • If the risk score is high enough, the system may decline the payment, ask the cardholder to confirm it, or route the case to a fraud analyst.
  • The tradeoff is simple to describe and hard to manage: every missed fraud loses money, while every false positive interrupts a legitimate customer.
  • Those decisions usually have to be made under tight latency limits, so the system has to be fast enough for the payment flow and accurate enough to justify intervention.

3.4 Class Imbalance in Fraud Detection

Fraud is rare in most transaction streams. The harder issue is that the non-fraud class is not clean or homogeneous: it includes routine purchases, travel spending, seasonal spikes, emergency purchases, and other behavior that is unusual but still legitimate.

Characteristics in Fraud Data

  • Rarity of Fraud: Fraudulent transactions are usually a small fraction (often \(< 1\%\), sometimes \(< 0.1\%\)) of total transactions.

  • Standard accuracy can be misleading; a model that predicts “no fraud” for every transaction can still look excellent on accuracy alone.

Impact of Sample Size

The difficulty of modeling depends on both the imbalance ratio (IR) and the absolute number of fraud cases.

A higher imbalance ratio (IR) indicates greater class imbalance: \[IR = \frac{\text{Number of samples in majority class}}{\text{Number of samples in minority class}}\]

The absolute number of samples in the minority class (fraud) also matters:

  • Large Datasets: A low fraud rate (e.g., 0.1%) in a large dataset (e.g., 100 million transactions) can still yield enough fraud examples (e.g., 100,000) to train and test a model sensibly.

  • Small Datasets: The same low fraud rate in a small dataset (e.g., 1,000 transactions) leaves you with very few fraud examples (approximately 1 fraud case if the rate is 0.1%). That is usually not enough for reliable training or evaluation.

In the rest of the lab, assume a large dataset with enough fraud cases to fit models, but still with the severe class imbalance seen in production.

4 Data Prep for Big Data

  • The simulated data is stored one day per parquet file, which is close to how transaction feeds are often archived in practice.

  • pandas is still useful for inspection and model interfaces, but it is not the best default for every heavy data step in this lab.
  • That is why the workflow leans on polars and DuckDB: they read parquet well and handle large aggregations more comfortably.

4.3 Prequential Data Splitting (using Polars)

To properly evaluate fraud detection models, we split the data chronologically into training, tuning (calibration), and test sets based on the TX_DATETIME column.

Also known as out-of-time validation, prequential data splitting ensures that future transactions do not influence model training or validation.

The lab splits the data into three sets:

  • df_train: Transactions before 2018-06-30 (inclusive).
  • df_tuning: Transactions between 2018-07-01 (inclusive) and 2018-08-31 (inclusive).
  • df_test: Transactions after 2018-09-01 (inclusive).
Code 
# Prequential data splitting using Polars

def split_data(df: pl.DataFrame
            , type: str = "train") -> pl.DataFrame:
    """
    Splits the data into train, tuning, and test sets based on TX_DATETIME.
    """
    if type == "train":
        return df.filter(pl.col("TX_DATETIME") <= pl.date(2018, 6, 30))
    elif type == "tuning":
        return df.filter((pl.col("TX_DATETIME") >= pl.date(2018, 7, 1)) 
                        & (pl.col("TX_DATETIME") <= pl.date(2018, 8, 31)))
    elif type == "test":
        return df.filter(pl.col("TX_DATETIME") >= pl.date(2018, 9, 1))
    else:
        raise ValueError(f"Unknown split type: {type}")

6 Feature Engineering

Single transactions do not carry enough context on their own. Fraud detection gets stronger once the model can see what the customer and the terminal have been doing over the last 1, 7, and 30 days.

The feature set follows Chapter 3 of Le Borgne et al. (2022), but the implementation is rewritten with Polars expressions so the rolling-window calculations stay practical on a larger dataset.

6.1 Polars Feature Functions

We will create functions to generate the following features. Each function will take a Polars DataFrame as input and return a Polars DataFrame with the new feature(s) added.

Date-related features:

  1. TX_DURING_WEEKEND: whether the transaction occurred on a weekend (1 if weekend, 0 if weekday).

  2. TX_DURING_NIGHT: whether the transaction occurred during the night (1 if night, 0 if day).

Customer features:

  1. CUSTOMER_ID_NB_TX_1DAY_WINDOW: number of transactions for the customer in the last 1 day.

  2. CUSTOMER_ID_AVG_AMOUNT_1DAY_WINDOW: average transaction amount for the customer in the last 1 day.

  3. CUSTOMER_ID_NB_TX_7DAY_WINDOW: number of transactions for the customer in the last 7 days.

  4. CUSTOMER_ID_AVG_AMOUNT_7DAY_WINDOW: average transaction amount for the customer in the last 7 days.

  5. CUSTOMER_ID_NB_TX_30DAY_WINDOW: number of transactions for the customer in the last 30 days.

  6. CUSTOMER_ID_AVG_AMOUNT_30DAY_WINDOW: average transaction amount for the customer in the last 30 days.

Terminal features:

  1. TERMINAL_ID_NB_TX_1DAY_WINDOW: number of transactions for the terminal in the last 1 day. Lagged 7 days to account for a lag in fraud labeling.

  2. TERMINAL_ID_RISK_1DAY_WINDOW: proportion of fraudulent transactions for the terminal in the last 1 day. Lagged 7 days to account for a lag in fraud labeling.

  3. TERMINAL_ID_NB_TX_7DAY_WINDOW: number of transactions for the terminal in the last 7 days. Lagged 7 days to account for a lag in fraud labeling.

  4. TERMINAL_ID_RISK_7DAY_WINDOW: proportion of fraudulent transactions for the terminal in the last 7 days. Lagged 7 days to account for a lag in fraud labeling.

  5. TERMINAL_ID_NB_TX_30DAY_WINDOW: number of transactions for the terminal in the last 30 days. Lagged 7 days to account for a lag in fraud labeling.

  6. TERMINAL_ID_RISK_30DAY_WINDOW: proportion of fraudulent transactions for the terminal in the last 30 days. Lagged 7 days to account for a lag in fraud labeling.

6.1.1 Weekend Flag

The feature indicates if a transaction occurred on a weekend.

Code 
def fe_is_weekend(df: pl.DataFrame) -> pl.DataFrame:
    """Adds IS_WEEKEND feature (1 if weekend, 0 if weekday)."""
    # Polars weekday(): Monday=1, Sunday=7. So, Saturday (6) or Sunday (7) are weekends.
    return df.with_columns(
        TX_DURING_WEEKEND = (pl.col("TX_DATETIME").dt.weekday()
                                .is_in([6, 7])
                                .cast(pl.UInt8))
    )

6.1.2 Night Flag

The feature indicates if a transaction occurred during the night (defined as between midnight and 6:00 AM).

Code 
def fe_is_night(df: pl.DataFrame) -> pl.DataFrame:
    """Adds IS_NIGHT feature (1 if hour < 6, 0 otherwise)."""
    return df.with_columns(
        TX_DURING_NIGHT = pl.when(pl.col("TX_DATETIME").dt.hour() < 6)
                            .then(1)
                            .otherwise(0)
                            .cast(pl.UInt8)
    )

6.1.3 Customer Features

The features describe a customer’s recent transaction count and average transaction amount over different rolling time windows (1, 7, and 30 days).

Code 
def fe_customer_spending(df: pl.DataFrame, window_sizes_in_days=[1, 7, 30]) -> pl.DataFrame:
    """
    Calculates customer spending features:
    - Count of transactions per customer over different window_sizes.
    - Average transaction amount per customer over different window_sizes.
    """

    df_int = (
        df
        .sort(pl.col("CUSTOMER_ID"), pl.col("TX_DATETIME")) # very important to sort
        .with_columns(
            pl.col("CUSTOMER_ID").cast(pl.UInt64),
            pl.lit(1).alias("TX_COUNT")
        )
    )

    result_df = df_int
    for window in window_sizes_in_days:
        result_df = (
            result_df
            .with_columns(
                pl.col("TX_COUNT")
                    .rolling_sum_by(by=pl.col("TX_DATETIME")
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("CUSTOMER_ID")
                .alias(f"CID_NB_TX_{window}DAY_WINDOW"),

                pl.col("TX_AMOUNT")
                    .rolling_mean_by(by=pl.col("TX_DATETIME")
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("CUSTOMER_ID")
                .alias(f"CID_AVG_AMOUNT_{window}DAY_WINDOW")
            )
        )

    return result_df

6.1.4 Terminal Features

The features describe a point-of-sale terminal’s recent transactions count and the proportion of fraudulent transactions over several rolling time windows (1, 7, 30 days). A 7-day delay period is introduced because fraud labels are not be available immediately.

Code 
def fe_terminal_risk(
    df: pl.DataFrame, 
    delay_period_days: int = 7, 
    window_sizes_in_days: list = [1, 7, 30]
) -> pl.DataFrame:
    """
    Adds terminal risk features for various window sizes, considering a 7-day delay in fraud labeling.
    For each window, computes:
        - Number of transactions at the terminal in the window (lagged by delay_period_days)
        - Proportion of fraudulent transactions at the terminal in the window (lagged)
    """
    # Sort by TERMINAL_ID and TX_DATETIME for correct rolling calculations
    df_int = (
        df
        .sort(pl.col("TERMINAL_ID")
            , pl.col("TX_DATETIME")
        )
        .with_columns(
            pl.col("TERMINAL_ID").cast(pl.UInt64),
            pl.lit(1).alias("TX_COUNT")
        )
    )

    result_df = df_int
    for window in window_sizes_in_days:

        full_offset = window + delay_period_days

        # Rolling window with delay (offset)
        result_df = (
            result_df
            .with_columns(
                pl.col("TX_COUNT")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_TX_{window}DAY_WINDOW"),

                pl.col("TX_COUNT")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{full_offset}d"
                        , closed="both"
                        )
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_TX_{full_offset}DAY_WINDOW"),


                pl.col("TX_FRAUD")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{window}d"
                        , closed="both"
                        )  
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_FRAUD_{window}DAY_WINDOW"),

                pl.col("TX_FRAUD")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{full_offset}d"
                        , closed="both"
                        )  
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_FRAUD_{full_offset}DAY_WINDOW")
            )

        )

        # Compute risk (proportion of frauds)
        risk_col = f"TID_RISK_{window}DAY_WINDOW"

        nb_tx_col = f"TID_NB_TX_{window}DAY_WINDOW"
        nb_fraud_col = f"TID_NB_FRAUD_{window}DAY_WINDOW"

        nb_tx_col_fulloffset = f"TID_NB_TX_{full_offset}DAY_WINDOW"
        nb_fraud_col_fulloffset = f"TID_NB_FRAUD_{full_offset}DAY_WINDOW"

        result_df = (
            result_df
            .with_columns(
                (pl.col(nb_tx_col_fulloffset) - pl.col(nb_tx_col)).alias(nb_tx_col),
                (pl.col(nb_fraud_col_fulloffset) - pl.col(nb_fraud_col)).alias(nb_fraud_col)
            )
            .select(pl.col("*")
                .exclude([nb_tx_col_fulloffset, nb_fraud_col_fulloffset])
                )
            .with_columns(
                pl.when(pl.col(nb_tx_col) > 0)
                    .then(pl.col(nb_fraud_col) / (pl.col(nb_tx_col)))
                    .otherwise(0.0)
                .alias(risk_col)
            )

        )

        # Drop the intermediate fraud count column
        result_df = result_df.drop(nb_fraud_col)

    return result_df

def extract_positive_class_scores(probabilities) -> np.ndarray:
    if isinstance(probabilities, pd.DataFrame):
        if 1 in probabilities.columns:
            return probabilities[1].to_numpy(dtype=float)
        return probabilities.iloc[:, -1].to_numpy(dtype=float)

    probabilities = np.asarray(probabilities, dtype=float)
    if probabilities.ndim == 1:
        return probabilities
    return probabilities[:, -1]


def predict_positive_class_scores(estimator, features) -> np.ndarray:
    return extract_positive_class_scores(estimator.predict_proba(features))


def find_best_threshold(y_true, y_score, num_thresholds: int = 200) -> float:
    thresholds = np.linspace(0.005, 0.995, num_thresholds)
    best_threshold = 0.5
    best_f1 = -1.0

    for threshold in thresholds:
        y_pred = (y_score >= threshold).astype(int)
        current_f1 = f1_score(y_true, y_pred, zero_division=0)
        if current_f1 > best_f1:
            best_f1 = current_f1
            best_threshold = threshold

    return float(best_threshold)


def expected_calibration_error(
    y_true,
    y_prob,
    n_bins: int = 20,
    binning: str = "uniform"
) -> float:
    y_true = np.asarray(y_true, dtype=float).reshape(-1)
    y_prob = extract_positive_class_scores(y_prob).reshape(-1)

    if y_true.shape[0] != y_prob.shape[0]:
        raise ValueError("y_true and y_prob must have the same number of observations.")
    if n_bins < 1:
        raise ValueError("n_bins must be at least 1.")
    if np.any((y_prob < 0) | (y_prob > 1)):
        raise ValueError("y_prob must contain probabilities in [0, 1].")

    if binning == "uniform":
        bin_edges = np.linspace(0.0, 1.0, n_bins + 1)
    elif binning == "quantile":
        bin_edges = np.quantile(y_prob, np.linspace(0.0, 1.0, n_bins + 1))
        bin_edges[0] = 0.0
        bin_edges[-1] = 1.0
        bin_edges = np.unique(bin_edges)
        if len(bin_edges) < 2:
            return float(abs(y_true.mean() - y_prob.mean()))
    else:
        raise ValueError("binning must be either 'uniform' or 'quantile'.")

    bin_ids = np.digitize(y_prob, bin_edges[1:-1], right=False)
    ece = 0.0

    for bin_id in range(len(bin_edges) - 1):
        mask = bin_ids == bin_id
        if not np.any(mask):
            continue
        ece += mask.mean() * abs(y_true[mask].mean() - y_prob[mask].mean())

    return float(ece)


def _coerce_results_frame(results) -> pd.DataFrame:
    if isinstance(results, pd.DataFrame):
        return results
    if isinstance(results, pd.Series):
        return pd.DataFrame([results.to_dict()])
    if isinstance(results, dict):

def _select_single_result_row(results, filters: dict | None = None) -> pd.Series:
    results_df = _coerce_results_frame(results)

    if filters:
        mask = pd.Series(True, index=results_df.index)
        for column, value in filters.items():
            if column not in results_df.columns:
                raise KeyError(f"Column '{column}' not found in results.")
            mask &= results_df[column].eq(value)
        results_df = results_df.loc[mask]

    if results_df.empty:
        raise ValueError("No rows matched the requested filters.")
    if len(results_df) > 1:
        raise ValueError("Multiple rows matched the requested filters. Add more specific filters.")

    return results_df.iloc[0]


def _extract_metric_value_from_row(row: pd.Series, metric_name: str) -> float:

    value = row[metric_name]

        raise ValueError(f"Metric '{metric_name}' is missing.")

def _compare_metric_values(
    left_value: float,
    right_value: float,
    better_when: str = "higher",
    tolerance: float = 1e-4
) -> str:
    if better_when not in {"higher", "lower"}:
        raise ValueError("better_when must be either 'higher' or 'lower'.")

    if abs(left_value - right_value) <= tolerance:
        return "tie"

    if better_when == "higher":
        return "left" if left_value > right_value else "right"

    return "left" if left_value < right_value else "right"


def _format_metric_value(value: float, decimals: int = 4) -> str:
    return f"{float(value):.{decimals}f}"

def _resolve_metric_direction(metric_name: str, better_when: dict[str, str] | None = None) -> str:

        "ece"

def build_strategy_metric_comparison_text(

                f"{right_label} is better on {metric_label} "
                f"({_format_metric_value(right_value)} versus {_format_metric_value(left_value)})."
            )

8.2 Do-Nothing Approach

Rationale:

  • The observed class imbalance is part of the business problem, not automatically a defect in the data.
  • Before changing the data, it is worth asking how far a strong learner can get with the original base rate.
  • Many modern machine learning estimators can still rank rare fraud cases reasonably well if the features are informative and the evaluation setup is sensible, including:
    • Tree-based ensembles (LightGBM, XGBoost, CatBoost)
    • Neural networks
  • This baseline matters because it tells us whether later imbalance-handling tricks are actually helping or just making the workflow more complicated.

8.3 Class-Sensitive Evaluation Approach

Rationale:

  • In fraud detection, the positive class (fraud) is much rarer than the negative class (legitimate transactions).
  • ROC-AUC is still a useful broad, threshold-free ranking and discrimination diagnostic:
    • ROC-AUC summarizes how well the model ranks fraud above legitimate transactions across all possible decision thresholds.
    • It is familiar, easy to benchmark, and helpful for comparing broad ranking quality across challenger models.
    • However, it should not be the primary or sole success metric in highly imbalanced fraud detection.
    • ROC plots the True Positive Rate (TPR, recall) against the False Positive Rate (FPR) at different decision thresholds.
    • FPR is calculated as \(\frac{\text{False Positives}}{\text{All Actual Negatives}}\). When the legitimate class is very large, even a low FPR can still correspond to an operationally unacceptable number of false alerts.
  • Precision-Recall (PR) curves are often better aligned with rare-event fraud operations:
    • The PR curve plots Precision (\(\frac{\text{TP}}{\text{TP} + \text{FP}}\)) against Recall (\(\frac{\text{TP}}{\text{TP} + \text{FN}}\)).
    • Precision directly measures how many of the predicted frauds are actually fraud, which is critical when investigating alerts.
    • PR curves do not account for true negatives, making them more sensitive to the model’s performance on the minority class.
  • PR-AUC (Area Under the PR Curve, also called Average Precision or AP) summarizes the PR curve in a single number.
    • A high PR-AUC means the model can achieve both high precision and high recall for fraud detection.
    • The baseline (random classifier) PR-AUC equals the prevalence of the positive class, making it easier to assess model quality.
  • Operational metrics still matter for deployment:
    • Precision at top-\(k\) or top-\(x\%\)
    • Recall at fixed review capacity
    • Dollar-weighted capture or savings
    • False positives per true positive
    • Calibration diagnostics

PR-AUC and AP are conceptually similar, but AP calculates the area under the PR curve using the observed precision values at each threshold rather than an interpolated curve. Industry practice is not “never use ROC-AUC”; rather, ROC-AUC is a useful secondary diagnostic but is insufficient on its own for highly imbalanced fraud problems. In this lab, we use AP (average_precision) as the primary model-selection metric while keeping ROC-AUC as a secondary ranking diagnostic. For details, please refer to the scikit-learn documentation on Average Precision.

Process:

  1. Train multiple estimators on the original, imbalanced dataset.
  2. Select the best model using average precision as the primary metric while reviewing ROC-AUC as a secondary threshold-free diagnostic.
  3. Check for proper probability calibration. Calibrate the predicted probabilities if needed.
  4. Use the tuning set to select an optimal decision threshold with operational metrics in mind (for example, F1-score, precision at top-\(k\), recall at fixed review capacity, false positives per true positive, or dollar-weighted capture).
  5. Evaluate the performance of the calibrated model and chosen threshold on the test set using AP, ROC-AUC, and the operational metrics most relevant to the fraud workflow.

8.4 Cost-Sensitive Learning Approach

Rationale:

  • Cost-sensitive learning assigns different costs to classification errors.
  • In fraud detection, failing to detect a fraudulent transaction (a false negative) is typically much more costly than incorrectly flagging a legitimate transaction as fraudulent (a false positive).
  • By incorporating these costs, the model is encouraged to pay more attention to the minority (fraud) class.

Implementation Detail:

  • One common way to implement cost-sensitive learning is by using sample weights:
    • Instances from the minority class are given higher weights.
    • Instances from the majority class are given lower weights.
    • The weights are inversely proportional to class frequencies.
  • AutoGluon lets us do this without hand-building a separate weight column:
    • Set the sample_weight argument in predictor_args to the special string "balance_weight".
    • AutoGluon then computes inverse-frequency weights and applies them during training.
  • Sample weights are not needed for the test set evaluation.

Process:

  1. Specify sample weights for each observation in the training set by setting sample_weight to "balance_weight".
  2. Train multiple estimators on the original, imbalanced dataset.
  3. Select the best model with the highest average precision on the test set.
  4. Check for proper probability calibration. Calibrate the predicted probabilities if needed.
  5. Select the optimal decision threshold.
  6. Evaluate the performance of the calibrated model and the optimal threshold on the test set.

8.5 Resampling Approach

Rationale:

  • Class-based resampling techniques modify the training data’s class distribution to create a less extreme dataset for model fitting.
  • Resampling is performed only on the training data.
  • The tuning and test sets remain in their original, imbalanced form to reflect reality (i.e., fraud is rare).
  • In fraud detection, the model still needs plenty of legitimate transactions in view. The job is to learn fraud patterns without hiding the wide range of legitimate behavior that can look suspicious at first glance.

A fully balanced 1:1 sample is often a bad default for fraud detection because it creates a training world the business never sees in production. Fraud is rare, legitimate traffic is abundant, and some legitimate customers will always look odd in the raw data.

Why 1:1 is usually a poor statistical default

  • It throws away too much useful non-fraud information. Legitimate transactions contain customer, merchant, device, geographic, seasonal, and amount variation that helps define the real non-fraud population.
  • It can inflate false positives because the sampled data teaches the model that fraud is much more common than it really is unless later calibration, weighting, or prior correction is applied.
  • It damages probability calibration. The rank ordering may still be useful, but raw predicted fraud probabilities from a 1:1 sample are usually not business-usable without correction.
  • More non-fraud controls usually improve precision because they help the model distinguish fraud from legitimate weirdness. The gains are often largest when moving from 1:1 toward ratios such as 5:1 or 10:1.
  • It can make the model learn sampling artifacts if the undersampled non-fraud group no longer looks like the real production population.

Why 1:1 is usually a poor business default

  • False positives are expensive: they can create declined legitimate transactions, manual review cost, customer frustration, merchant friction, and lost revenue.
  • The business does not operate in a 50% fraud world. The model must prioritize the riskiest cases inside a large sea of legitimate activity.
  • Fraud teams have limited alert capacity, so the model must rank cases well on a realistic population rather than on an artificially balanced slice.
  • Many legitimate transactions look suspicious. The business wants to separate fraud from legitimate weirdness, not just from a tiny puddle of non-fraud examples.

A practical compromise is to reduce the imbalance without pretending fraud and non-fraud are equally common:

Non-fraud:Fraud ratio Why it is useful
5:1 Useful when fraud cases are limited and model training cost matters, but it is still a relatively aggressive rebalance.
10:1 Strong general-purpose default for rare-event modeling.
20:1 Better when legitimate behavior is highly diverse and cheap to include.
1:1 Usually too artificial for deployment-oriented fraud modeling; keep it as a classroom simplification rather than a default.

Common Resampling Schemes:

  • Random Undersampling (RUS): Randomly remove instances from the majority class until the class imbalance is less extreme. In practice, a moderate target such as 5:1, 10:1, or 20:1 is often preferable to forcing a 1:1 sample. This can lead to loss of information from the majority class. However, the scheme substantially reduces the dataset size, which can be beneficial for training speed and memory usage.
  • Random Oversampling (ROS): Randomly duplicate instances from the minority class. Here too, a moderate target such as 5:1, 10:1, or 20:1 is often safer than a 1:1 sample. This can lead to overfitting on the minority class examples.
  • SMOTE (Synthetic Minority Over-sampling Technique): Creates synthetic samples for the minority class by interpolating between actual minority instances. This can help to create a more diverse set of minority examples than simple oversampling.
  • Hybrid Methods: Combine oversampling and undersampling. For example, first apply SMOTE to generate synthetic samples for the minority class, then apply random undersampling to the majority class.

Process:

  1. Use RUS, ROS, SMOTE, or hybrid to create a less extreme training set. For RUS and ROS, ratios such as 5:1, 10:1, or 20:1 non-events to events are often more defensible than a fully balanced 1:1 sample.
  2. Train multiple estimators on the resampled training set.
  3. Select the best model with the highest average precision on the test set (which is imbalanced).
  4. Check for proper probability calibration. Calibrate the predicted probabilities if needed.
  5. Select the optimal decision threshold.
  6. Evaluate the performance of the calibrated model and the optimal threshold on the test set (which is imbalanced).

8.5.6 Prior Correction for Logistic Regression After Class-Based Resampling

When a logistic regression is fit on a class-resampled version of the training set, the fitted probabilities live on the resampled class-prior scale rather than the original training-set scale. For random undersampling or random oversampling, if the sampling depends only on the class label and does not otherwise change the conditional feature distributions \(P(X \mid Y=1)\) and \(P(X \mid Y=0)\), then the slope coefficients are preserved and only the intercept changes.

Let

  • \(\hat\pi_\text{original}\) = prior_original, the fraud rate in the original non-resampled training set.
  • \(\hat\pi_\text{sampled}\) = prior_resampled, the fraud rate in the class-resampled training set.
  • \(\widehat{\alpha}_\text{sampled}\) = intercept_hat_resampled, the intercept from a logistic regression fit on the sampled training set.
  • \(\widehat{\beta}_\text{sampled}\) = the slope vector from the same sampled-data fit.

Write the sampled-data linear predictor as

\[ \widehat{\eta}_\text{sampled}(x) = \widehat{\alpha}_\text{sampled} + x^\top \widehat{\beta}_\text{sampled}. \]

If class-1 observations are retained or duplicated by a class-specific factor \(c_1\) and class-0 observations by a factor \(c_0\), then class-based sampling implies

\[ \frac{P_\text{sampled}(Y=1 \mid X=x)}{P_\text{sampled}(Y=0 \mid X=x)} = \frac{c_1}{c_0} \cdot \frac{P_\text{original}(Y=1 \mid X=x)}{P_\text{original}(Y=0 \mid X=x)}. \]

Therefore the sampled and original log-odds differ by a constant:

\[ \widehat{\eta}_\text{original}(x) = \widehat{\eta}_\text{sampled}(x) - \log\left(\frac{c_1}{c_0}\right). \]

The same constant can be written in terms of the two marginal class odds because

\[ \log\left(\frac{c_1}{c_0}\right) = \operatorname{logit}(\hat\pi_\text{sampled}) - \operatorname{logit}(\hat\pi_\text{original}). \]

Substituting gives the post-fit prior correction:

\[ \widehat{\eta}_\text{original}(x) = \widehat{\eta}_\text{sampled}(x) + \operatorname{logit}(\hat\pi_\text{original}) - \operatorname{logit}(\hat\pi_\text{sampled}). \]

Equivalently, only the intercept changes:

\[ \widehat{\alpha}_\text{original} = \widehat{\alpha}_\text{sampled} + \operatorname{logit}(\hat\pi_\text{original}) - \operatorname{logit}(\hat\pi_\text{sampled}), \qquad \widehat{\beta}_\text{original} = \widehat{\beta}_\text{sampled}. \]

The corrected fitted probability on the original training-set prior scale is

\[ \widehat{P}_\text{original}(Y=1 \mid X=x) = \frac{1}{1+\exp\{-\widehat{\eta}_\text{original}(x)\}}. \]

For fraud data, \(\hat\pi_\text{original}\) is usually much smaller than \(\hat\pi_\text{sampled}\). Therefore the correction term is usually negative, lowering the intercept and pulling predicted probabilities back toward the original non-resampled training-set fraud rate.

Do not use sklearn.linear_model.LogisticRegression for this demonstration. Its default fit uses L2 regularization and the LBFGS solver, whereas the derivation here is meant to align with an unpenalized binomial GLM fit with statsmodels via IRLS.

This correction fixes the prior shift induced by class-based sampling and maps the fitted model back to the original training-set prior. It is not the same as a nonparametric calibration step such as isotonic regression. If the deployment fraud rate differs from the original training-set fraud rate, replace prior_original with the desired deployment prior. The correction is exact only when resampling changes class frequencies without otherwise changing \(P(X \mid Y)\). For SMOTE or feature-dependent sampling, treat the same intercept shift as an approximation and validate on untouched data.

8.5.7 ROS Prior-Correction for Logistic Regression

The same prior-correction applies to random oversampling. Because RandomOverSampler duplicates minority-class observations without changing their feature values, the intercept-shift correction is the right adjustment when the goal is to recover probabilities on the original training-set class-prior scale.

The same warning applies here: do not use sklearn.linear_model.LogisticRegression for this example. Its default L2-penalized LBFGS fit is not the same object as the unpenalized binomial GLM fitted with statsmodels via IRLS.

As with RUS, this correction addresses the prior shift introduced by class-based oversampling. It is still a prior correction rather than a separate calibration model. Because ROS duplicates observed minority cases rather than synthesizing new feature vectors, the intercept shift is on firmer ground here than it is for SMOTE.

Template For Interpreting Your Results:

Do not copy the discussion below verbatim into a submission. Use it as a guide for writing up what your own runs actually showed.

  • Strategy 1 (Do Nothing / Original Imbalanced Data): Treat this as the control case. If it already ranks fraud reasonably well, the feature set is doing more work than you might expect. If ROC-AUC looks respectable but thresholded performance is weak, that is a reminder that broad ranking quality does not automatically translate into a workable alert rule.

  • Strategy 2 (Class-Sensitive Metric Selection): Using PR-AUC (average_precision) as the primary metric usually lines up better with how fraud teams work, because only a small high-risk slice of transactions will be reviewed. Keep ROC-AUC in the report, but let AP and threshold metrics carry more weight in the decision.

  • Strategy 3 (Cost-Sensitive Learning with balance_weight): When you have enough data, reweighting is often the least disruptive way to make the learner take fraud more seriously. It keeps the original sample intact and usually creates fewer side effects than aggressive resampling.

  • Strategy 4 (Resampling — RUS, ROS, SMOTE, Hybrid): This family asks a practical question: if you are going to rebalance, how far should you go? For fraud data, 1:1 is often too aggressive. Ratios such as 5:1, 10:1, and 20:1 usually preserve more legitimate behavior while still making the fit easier:

    • RUS (Random Undersampling): RUS lowers training cost by throwing away majority-class rows, but the discarded rows may contain exactly the legitimate variation you need.
    • ROS (Random Oversampling): ROS is easy to implement, but duplicated fraud cases can make the model overreact to a small set of examples.
    • SMOTE: SMOTE creates synthetic minority samples by interpolation. It can add variety relative to ROS, but it still changes the training distribution and usually hurts raw calibration.
    • Hybrid (SMOTE + RUS): Hybrid pipelines can be effective, but they buy flexibility by adding more moving parts to debug and recalibrate.
    • Probability Calibration: Any resampling scheme can distort score calibration. If the probabilities will be used as probabilities rather than just rankings, recalibrate on an untouched tuning set.
    • Performance: Do not assume resampling wins by default. In many fraud problems, better features, a better selection metric, or class weights buy more than a more complex sampling pipeline.

General Recommendations:

  1. Do Not Rely on One Metric: In imbalanced fraud detection, ROC-AUC is a useful secondary view of ranking quality, not the whole report card. Pair it with PR-AUC (average_precision) and operations-aware metrics such as precision at top-\(k\)/top-\(x\%\), recall at fixed review capacity, dollar-weighted capture or savings, false positives per true positive, and calibration diagnostics.

  2. Try Cost-Sensitive Learning Before Aggressive Resampling: If you have enough data and want to avoid synthetic examples or heavy undersampling, sample_weight='balance_weight' is often the cleanest next step.

  3. Resample Carefully, Not Automatically: Do not manipulate training data (e.g., via RUS, ROS, SMOTE) solely because it is imbalanced or feels difficult to model. Such methods can introduce their own problems:

    • Use Moderate Targets: For RUS and ROS, ratios such as 5:1, 10:1, or 20:1 non-events to events are often more realistic than 1:1.
    • Preserve Legitimate Variation: More non-fraud controls help the model learn the sea of legitimate activity rather than a tiny puddle of non-fraud examples.
    • Loss of Information (RUS): Undersampling can discard valuable data from the majority class, especially if you force the data all the way to 1:1.
    • Overfitting (ROS, SMOTE): Oversampling can lead to models that are too specific to the (potentially synthetic) minority samples, and a 1:1 ROS sample makes that risk worse.
    • Avoid False-Positive Inflation: A fraud-heavy sampled training set can teach the model a world where fraud is much more common than it really is, which may translate into more alerts than the business can absorb.
    • Destroyed Probability Calibration: This is a major issue, especially if the model outputs are used for more than just binary classification (e.g., risk scoring, estimating expected fraud rates). More aggressive rebalancing usually makes calibration harder.
    • Increased Complexity: More complex pipelines can be harder to maintain, understand, and debug.

  4. Calibrate If Scores Will Be Used as Probabilities: If the model scores feed queue priority, expected-loss estimates, or any downstream decision that depends on the size of the probability, good ranking is not enough. Calibrate on untouched data that reflects the real production population.

  5. Tune Thresholds on a Separate Set: Regardless of the imbalance strategy, always use a dedicated tuning set to find an optimal decision threshold that aligns with business objectives (e.g., maximizing F1-score, or balancing precision and recall according to specific cost considerations).

  6. Spend Time on Features: In fraud work, a small set of good recency, customer-history, and terminal-history features usually matters more than switching from one imbalance trick to another.

# Credit Card Fraud Detection

Fraud modeling is hard for a practical reason: the transactions you care about are rare, the losses can be large, and many legitimate transactions also look irregular in the raw data. Travel, holiday spending, or an unusually large purchase can resemble fraud unless the model has enough context to tell the difference.

## Card-present frauds

-   Card-present (CP) fraud occurs when a transaction is made using a **physical credit card** that has been lost, stolen, counterfeited, or intercepted.
-   The legitimate cardholder is typically unaware of the fraudulent use until they notice unauthorized transactions on their statement or are alerted by their bank.
-   What makes CP fraud distinct is that the **physical card** is used at the point of sale (e.g., a retail store or restaurant).

## Card-not-present frauds

-   Card-not-present (CNP) fraud occurs when credit card information (such as card number, expiry date, CVV) is used **without** the physical card being present.
-   This type of fraud is common in online purchases.
-   Fraudsters may obtain card details through data breaches, phishing scams, or malware.
-   CNP fraud has become increasingly prevalent with the growth of e-commerce.

## Credit Card Fraud Detection Systems (CCFDS)

-   **Credit Card Fraud Detection Systems (CCFDS)** score transactions in real-time or near real-time using signals such as amount, merchant or terminal history, timing, location, and recent customer activity.
-   If the risk score is high enough, the system may decline the payment, ask the cardholder to confirm it, or route the case to a fraud analyst.
-   The tradeoff is simple to describe and hard to manage: every missed fraud loses money, while every false positive interrupts a legitimate customer.
-   Those decisions usually have to be made under tight latency limits, so the system has to be fast enough for the payment flow and accurate enough to justify intervention.

## Class Imbalance in Fraud Detection

Fraud is rare in most transaction streams. The harder issue is that the non-fraud class is not clean or homogeneous: it includes routine purchases, travel spending, seasonal spikes, emergency purchases, and other behavior that is unusual but still legitimate.

**Characteristics in Fraud Data**

-   Rarity of Fraud: Fraudulent transactions are usually a small fraction (often $< 1\%$, sometimes $< 0.1\%$) of total transactions.

-   Standard accuracy can be misleading; a model that predicts "no fraud" for every transaction can still look excellent on accuracy alone.

**Impact of Sample Size**

The difficulty of modeling depends on both the imbalance ratio (IR) and the absolute number of fraud cases.

A higher imbalance ratio (IR) indicates greater class imbalance: $$IR = \frac{\text{Number of samples in majority class}}{\text{Number of samples in minority class}}$$

The absolute number of samples in the minority class (fraud) also matters:

-   **Large Datasets:** A low fraud rate (e.g., 0.1%) in a large dataset (e.g., 100 million transactions) can still yield enough fraud examples (e.g., 100,000) to train and test a model sensibly.

-   **Small Datasets:** The same low fraud rate in a small dataset (e.g., 1,000 transactions) leaves you with very few fraud examples (approximately 1 fraud case if the rate is 0.1%). That is usually not enough for reliable training or evaluation.

In the rest of the lab, assume a large dataset with enough fraud cases to fit models, but still with the severe class imbalance seen in production.

## Prequential Data Splitting (using Polars)

To properly evaluate fraud detection models, we split the data **chronologically** into training, tuning (calibration), and test sets based on the `TX_DATETIME` column.

Also known as out-of-time validation, prequential data splitting ensures that future transactions do not influence model training or validation.

The lab splits the data into three sets:

-   `df_train`: Transactions before `2018-06-30` (inclusive).
-   `df_tuning`: Transactions between `2018-07-01` (inclusive) and `2018-08-31` (inclusive).
-   `df_test`: Transactions after `2018-09-01` (inclusive).

# Feature Engineering

Single transactions do not carry enough context on their own. Fraud detection gets stronger once the model can see what the customer and the terminal have been doing over the last 1, 7, and 30 days.

The feature set follows Chapter 3 of @leborgne2022fraud, but the implementation is rewritten with Polars expressions so the rolling-window calculations stay practical on a larger dataset.

## Polars Feature Functions

We will create functions to generate the following features. Each function will take a Polars DataFrame as input and return a Polars DataFrame with the new feature(s) added.

Date-related features:

1.  TX_DURING_WEEKEND: whether the transaction occurred on a weekend (1 if weekend, 0 if weekday).

2.  TX_DURING_NIGHT: whether the transaction occurred during the night (1 if night, 0 if day).

Customer features:

3.  CUSTOMER_ID_NB_TX_1DAY_WINDOW: number of transactions for the customer in the last 1 day.

4.  CUSTOMER_ID_AVG_AMOUNT_1DAY_WINDOW: average transaction amount for the customer in the last 1 day.

5.  CUSTOMER_ID_NB_TX_7DAY_WINDOW: number of transactions for the customer in the last 7 days.

6.  CUSTOMER_ID_AVG_AMOUNT_7DAY_WINDOW: average transaction amount for the customer in the last 7 days.

7.  CUSTOMER_ID_NB_TX_30DAY_WINDOW: number of transactions for the customer in the last 30 days.

8.  CUSTOMER_ID_AVG_AMOUNT_30DAY_WINDOW: average transaction amount for the customer in the last 30 days.

Terminal features:

9.  TERMINAL_ID_NB_TX_1DAY_WINDOW: number of transactions for the terminal in the last 1 day. Lagged 7 days to account for a lag in fraud labeling.

10. TERMINAL_ID_RISK_1DAY_WINDOW: proportion of **fraudulent** transactions for the terminal in the last 1 day. Lagged 7 days to account for a lag in fraud labeling.

11. TERMINAL_ID_NB_TX_7DAY_WINDOW: number of transactions for the terminal in the last 7 days. Lagged 7 days to account for a lag in fraud labeling.

12. TERMINAL_ID_RISK_7DAY_WINDOW: proportion of **fraudulent** transactions for the terminal in the last 7 days. Lagged 7 days to account for a lag in fraud labeling.

13. TERMINAL_ID_NB_TX_30DAY_WINDOW: number of transactions for the terminal in the last 30 days. Lagged 7 days to account for a lag in fraud labeling.

14. TERMINAL_ID_RISK_30DAY_WINDOW: proportion of **fraudulent** transactions for the terminal in the last 30 days. Lagged 7 days to account for a lag in fraud labeling.

### Customer Features

The features describe a customer's recent transaction count and average transaction amount over different rolling time windows (1, 7, and 30 days).

```{python}
#| label: feature-customer-spending
def fe_customer_spending(df: pl.DataFrame, window_sizes_in_days=[1, 7, 30]) -> pl.DataFrame:
    """
    Calculates customer spending features:
    - Count of transactions per customer over different window_sizes.
    - Average transaction amount per customer over different window_sizes.
    """

    df_int = (
        df
        .sort(pl.col("CUSTOMER_ID"), pl.col("TX_DATETIME")) # very important to sort
        .with_columns(
            pl.col("CUSTOMER_ID").cast(pl.UInt64),
            pl.lit(1).alias("TX_COUNT")
        )
    )

    result_df = df_int
    for window in window_sizes_in_days:
        result_df = (
            result_df
            .with_columns(
                pl.col("TX_COUNT")
                    .rolling_sum_by(by=pl.col("TX_DATETIME")
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("CUSTOMER_ID")
                .alias(f"CID_NB_TX_{window}DAY_WINDOW"),

                pl.col("TX_AMOUNT")
                    .rolling_mean_by(by=pl.col("TX_DATETIME")
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("CUSTOMER_ID")
                .alias(f"CID_AVG_AMOUNT_{window}DAY_WINDOW")
            )
        )

    return result_df
```

### Terminal Features

The features describe a point-of-sale terminal's recent transactions count and the proportion of fraudulent transactions over several rolling time windows (1, 7, 30 days). A 7-day delay period is introduced because fraud labels are not be available immediately.

```{python}
#| label: feature-terminal-risk
def fe_terminal_risk(
    df: pl.DataFrame, 
    delay_period_days: int = 7, 
    window_sizes_in_days: list = [1, 7, 30]
) -> pl.DataFrame:
    """
    Adds terminal risk features for various window sizes, considering a 7-day delay in fraud labeling.
    For each window, computes:
        - Number of transactions at the terminal in the window (lagged by delay_period_days)
        - Proportion of fraudulent transactions at the terminal in the window (lagged)
    """
    # Sort by TERMINAL_ID and TX_DATETIME for correct rolling calculations
    df_int = (
        df
        .sort(pl.col("TERMINAL_ID")
            , pl.col("TX_DATETIME")
        )
        .with_columns(
            pl.col("TERMINAL_ID").cast(pl.UInt64),
            pl.lit(1).alias("TX_COUNT")
        )
    )

    result_df = df_int
    for window in window_sizes_in_days:

        full_offset = window + delay_period_days

        # Rolling window with delay (offset)
        result_df = (
            result_df
            .with_columns(
                pl.col("TX_COUNT")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{window}d"
                        , closed="both"
                        )
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_TX_{window}DAY_WINDOW"),

                pl.col("TX_COUNT")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{full_offset}d"
                        , closed="both"
                        )
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_TX_{full_offset}DAY_WINDOW"),


                pl.col("TX_FRAUD")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{window}d"
                        , closed="both"
                        )  
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_FRAUD_{window}DAY_WINDOW"),

                pl.col("TX_FRAUD")
                    .rolling_sum_by("TX_DATETIME"
                        , window_size=f"{full_offset}d"
                        , closed="both"
                        )  
                    .over("TERMINAL_ID")
                .alias(f"TID_NB_FRAUD_{full_offset}DAY_WINDOW")
            )

        )

        # Compute risk (proportion of frauds)
        risk_col = f"TID_RISK_{window}DAY_WINDOW"

        nb_tx_col = f"TID_NB_TX_{window}DAY_WINDOW"
        nb_fraud_col = f"TID_NB_FRAUD_{window}DAY_WINDOW"

        nb_tx_col_fulloffset = f"TID_NB_TX_{full_offset}DAY_WINDOW"
        nb_fraud_col_fulloffset = f"TID_NB_FRAUD_{full_offset}DAY_WINDOW"

        result_df = (
            result_df
            .with_columns(
                (pl.col(nb_tx_col_fulloffset) - pl.col(nb_tx_col)).alias(nb_tx_col),
                (pl.col(nb_fraud_col_fulloffset) - pl.col(nb_fraud_col)).alias(nb_fraud_col)
            )
            .select(pl.col("*")
                .exclude([nb_tx_col_fulloffset, nb_fraud_col_fulloffset])
                )
            .with_columns(
                pl.when(pl.col(nb_tx_col) > 0)
                    .then(pl.col(nb_fraud_col) / (pl.col(nb_tx_col)))
                    .otherwise(0.0)
                .alias(risk_col)
            )

        )

        # Drop the intermediate fraud count column
        result_df = result_df.drop(nb_fraud_col)

    return result_df
```

```{python}
def extract_positive_class_scores(probabilities) -> np.ndarray:
    if isinstance(probabilities, pd.DataFrame):
        if 1 in probabilities.columns:
            return probabilities[1].to_numpy(dtype=float)
        return probabilities.iloc[:, -1].to_numpy(dtype=float)

    probabilities = np.asarray(probabilities, dtype=float)
    if probabilities.ndim == 1:
        return probabilities
    return probabilities[:, -1]


def predict_positive_class_scores(estimator, features) -> np.ndarray:
    return extract_positive_class_scores(estimator.predict_proba(features))


def find_best_threshold(y_true, y_score, num_thresholds: int = 200) -> float:
    thresholds = np.linspace(0.005, 0.995, num_thresholds)
    best_threshold = 0.5
    best_f1 = -1.0

    for threshold in thresholds:
        y_pred = (y_score >= threshold).astype(int)
        current_f1 = f1_score(y_true, y_pred, zero_division=0)
        if current_f1 > best_f1:
            best_f1 = current_f1
            best_threshold = threshold

    return float(best_threshold)


def expected_calibration_error(
    y_true,
    y_prob,
    n_bins: int = 20,
    binning: str = "uniform"
) -> float:
    y_true = np.asarray(y_true, dtype=float).reshape(-1)
    y_prob = extract_positive_class_scores(y_prob).reshape(-1)

    if y_true.shape[0] != y_prob.shape[0]:
        raise ValueError("y_true and y_prob must have the same number of observations.")
    if n_bins < 1:
        raise ValueError("n_bins must be at least 1.")
    if np.any((y_prob < 0) | (y_prob > 1)):
        raise ValueError("y_prob must contain probabilities in [0, 1].")

    if binning == "uniform":
        bin_edges = np.linspace(0.0, 1.0, n_bins + 1)
    elif binning == "quantile":
        bin_edges = np.quantile(y_prob, np.linspace(0.0, 1.0, n_bins + 1))
        bin_edges[0] = 0.0
        bin_edges[-1] = 1.0
        bin_edges = np.unique(bin_edges)
        if len(bin_edges) < 2:
            return float(abs(y_true.mean() - y_prob.mean()))
    else:
        raise ValueError("binning must be either 'uniform' or 'quantile'.")

    bin_ids = np.digitize(y_prob, bin_edges[1:-1], right=False)
    ece = 0.0

    for bin_id in range(len(bin_edges) - 1):
        mask = bin_ids == bin_id
        if not np.any(mask):
            continue
        ece += mask.mean() * abs(y_true[mask].mean() - y_prob[mask].mean())

    return float(ece)


def _coerce_results_frame(results) -> pd.DataFrame:
    if isinstance(results, pd.DataFrame):
        return results
    if isinstance(results, pd.Series):
        return pd.DataFrame([results.to_dict()])
    if isinstance(results, dict):

    return pd.DataFrame(results)


def _select_single_result_row(results, filters: dict | None = None) -> pd.Series:
    results_df = _coerce_results_frame(results)

    if filters:
        mask = pd.Series(True, index=results_df.index)
        for column, value in filters.items():
            if column not in results_df.columns:
                raise KeyError(f"Column '{column}' not found in results.")
            mask &= results_df[column].eq(value)
        results_df = results_df.loc[mask]

    if results_df.empty:
        raise ValueError("No rows matched the requested filters.")
    if len(results_df) > 1:
        raise ValueError("Multiple rows matched the requested filters. Add more specific filters.")

    return results_df.iloc[0]


def _extract_metric_value_from_row(row: pd.Series, metric_name: str) -> float:
    if metric_name not in row.index:
        raise KeyError(f"Metric '{metric_name}' not found in results.")

    value = row[metric_name]
    if pd.isna(value):
        raise ValueError(f"Metric '{metric_name}' is missing.")

    return float(value)


def _compare_metric_values(
    left_value: float,
    right_value: float,
    better_when: str = "higher",
    tolerance: float = 1e-4
) -> str:
    if better_when not in {"higher", "lower"}:
        raise ValueError("better_when must be either 'higher' or 'lower'.")

    if abs(left_value - right_value) <= tolerance:
        return "tie"

    if better_when == "higher":
        return "left" if left_value > right_value else "right"

    return "left" if left_value < right_value else "right"


def _format_metric_value(value: float, decimals: int = 4) -> str:
    return f"{float(value):.{decimals}f}"


def _build_result_label(
    strategy_name: str,
    filters: dict | None = None,
    label: str | None = None
) -> str:
    if label is not None:
        return label

    filters = filters or {}
    detail_values = [str(value) for column, value in filters.items() if column != "Strategy"]
    if detail_values:
        return f"{strategy_name} ({', '.join(detail_values)})"

    return strategy_name


def build_calibration_comparison_text(
    model_a_name: str,
    model_a_scores,
    model_b_name: str,
    model_b_scores,
    y_true,
    dataset_name: str,
    n_bins: int = 20,
    binning: str = "uniform",
    tolerance: float = 1e-4
) -> str:
    ece_a = expected_calibration_error(y_true, model_a_scores, n_bins=n_bins, binning=binning)
    ece_b = expected_calibration_error(y_true, model_b_scores, n_bins=n_bins, binning=binning)
    comparison = _compare_metric_values(ece_a, ece_b, better_when="lower", tolerance=tolerance)

    if comparison == "tie":
        verdict = f"{model_a_name} and {model_b_name} have similar calibration"
    elif comparison == "left":
        verdict = f"{model_a_name} is better calibrated than {model_b_name}"
    else:
        verdict = f"{model_b_name} is better calibrated than {model_a_name}"

    return (
        f"On the {dataset_name}, {verdict} by Expected Calibration Error "
        f"({model_a_name}: {_format_metric_value(ece_a)}, {model_b_name}: {_format_metric_value(ece_b)}). "
        f"Lower ECE is better. Binning: {binning} with {n_bins} bins."
    )

        tolerance=tolerance,
    )


def _resolve_metric_direction(metric_name: str, better_when: dict[str, str] | None = None) -> str:
    if better_when and metric_name in better_when:
        direction = better_when[metric_name]
    elif metric_name.lower() in {
        "log_loss",
        "loss",
        "rmse",
        "mae",
        "mse",
        "ece"
    }:
        direction = "lower"
    elif metric_name.lower() in {
        "roc_auc",
        "score_test",
        "average_precision",
        "ap",
        "f1",
        "f1-score",
        "precision",
        "recall"
    }:
        direction = "higher"
    else:
        raise ValueError(
            f"No comparison direction was provided for metric '{metric_name}'. "
            "Pass better_when explicitly for this metric."
        )

    if direction not in {"higher", "lower"}:
        raise ValueError("better_when values must be either 'higher' or 'lower'.")

    return direction


def build_threshold_comparison_text(
    default_results,
    optimized_results,
    model_name: str | None = None,
    dataset_name: str = "test set",
    metric_name: str = "f1",
    metric_label: str = "F1-score",
    model_column: str = "model",
    default_threshold: float = 0.5,
    optimized_threshold: float | None = None,
    tolerance: float = 1e-4
) -> str:
    filters = {model_column: model_name} if model_name is not None else None
    default_row = _select_single_result_row(default_results, filters=filters)
    optimized_row = _select_single_result_row(optimized_results, filters=filters)

    default_metric = _extract_metric_value_from_row(default_row, metric_name)
    optimized_metric = _extract_metric_value_from_row(optimized_row, metric_name)
    comparison = _compare_metric_values(
        default_metric,
        optimized_metric,
        better_when="higher",
        tolerance=tolerance
    )

    model_label = model_name or "the selected model"
    if comparison == "tie":
        verdict = f"threshold tuning does not materially change the {metric_label} for {model_label}"
    elif comparison == "right":
        verdict = f"threshold tuning improves the {metric_label} for {model_label}"
    else:
        verdict = f"the default threshold still gives the higher {metric_label} for {model_label}"

    default_sentence = (
        f"The default threshold of {default_threshold:.3f} gives {metric_label} "
        f"{_format_metric_value(default_metric)}."
    )
    if optimized_threshold is None:
        optimized_sentence = (
            f"The optimized threshold gives {metric_label} {_format_metric_value(optimized_metric)}."
        )
    else:
        optimized_sentence = (
            f"The optimized threshold of {optimized_threshold:.3f} gives {metric_label} "
            f"{_format_metric_value(optimized_metric)}."
        )

    return f"On the {dataset_name}, {verdict}. {default_sentence} {optimized_sentence}"


def build_strategy_metric_comparison_text(
    results,
    left_strategy: str,
    right_strategy: str,
    metrics: list[str],
    left_filters: dict | None = None,
    right_filters: dict | None = None,
    left_label: str | None = None,
    right_label: str | None = None,
    metric_labels: dict[str, str] | None = None,
    better_when: dict[str, str] | None = None,
    context_name: str = "the current summary",
    tolerance: float = 1e-4
) -> str:
    left_filters = {**(left_filters or {}), "Strategy": left_strategy}
    right_filters = {**(right_filters or {}), "Strategy": right_strategy}

    left_row = _select_single_result_row(results, filters=left_filters)
    right_row = _select_single_result_row(results, filters=right_filters)

    left_label = _build_result_label(left_strategy, filters=left_filters, label=left_label)
    right_label = _build_result_label(right_strategy, filters=right_filters, label=right_label)
    metric_labels = metric_labels or {}
    better_when = better_when or {}

    sentences = [f"In {context_name}, comparing {left_label} with {right_label}."]

    for metric_name in metrics:
        metric_label = metric_labels.get(metric_name, metric_name)
        left_value = _extract_metric_value_from_row(left_row, metric_name)
        right_value = _extract_metric_value_from_row(right_row, metric_name)
        metric_direction = _resolve_metric_direction(metric_name, better_when=better_when)
        comparison = _compare_metric_values(
            left_value,
            right_value,
            better_when=metric_direction,
            tolerance=tolerance
        )

        if comparison == "tie":
            sentences.append(
                f"{metric_label} is similar ({left_label}: {_format_metric_value(left_value)}, "
                f"{right_label}: {_format_metric_value(right_value)})."
            )
        elif comparison == "left":
            sentences.append(
                f"{left_label} is better on {metric_label} "
                f"({_format_metric_value(left_value)} versus {_format_metric_value(right_value)})."
            )
        else:
            sentences.append(
                f"{right_label} is better on {metric_label} "
                f"({_format_metric_value(right_value)} versus {_format_metric_value(left_value)})."
            )

    return " ".join(sentences)


def build_running_metric_row(
    strategy_name: str,
    model_name: str,
    y_tuning,
    tuning_scores,
    y_test,
    test_scores,
    best_threshold: float | None = None
) -> dict:
    tuning_scores = np.asarray(tuning_scores, dtype=float)
    test_scores = np.asarray(test_scores, dtype=float)

    if best_threshold is None:
        best_threshold = find_best_threshold(y_tuning, tuning_scores)

    y_pred_test = (test_scores >= best_threshold).astype(int)

    return {
        "Strategy": strategy_name,
        "Model": model_name,
        "Test ROC-AUC": roc_auc_score(y_test, test_scores),
        "Test Average Precision": average_precision_score(y_test, test_scores),
        "best_threshold": float(best_threshold),
        "precision": precision_score(y_test, y_pred_test, zero_division=0),
        "recall": recall_score(y_test, y_pred_test, zero_division=0),
        "F1-score": f1_score(y_test, y_pred_test, zero_division=0)
    }
```

## Do-Nothing Approach

**Rationale:**

-   The observed class imbalance is part of the business problem, not automatically a defect in the data.
-   Before changing the data, it is worth asking how far a strong learner can get with the original base rate.
-   Many modern machine learning estimators can still rank rare fraud cases reasonably well if the features are informative and the evaluation setup is sensible, including:
    -   Tree-based ensembles (LightGBM, XGBoost, CatBoost)
    -   Neural networks
-   This baseline matters because it tells us whether later imbalance-handling tricks are actually helping or just making the workflow more complicated.

**Process:**

1.  Train multiple estimators on the original, imbalanced dataset.
2.  Select the best model with the lowest **log loss** on the test set.
3.  Check for proper probability calibration. Calibrate the predicted probabilities if needed.
4.  Select the optimal decision threshold.
5.  Evaluate the performance of the calibrated model and the optimal threshold on the test set.

## Class-Sensitive Evaluation Approach

**Rationale:**

-   In fraud detection, the positive class (fraud) is much rarer than the negative class (legitimate transactions).
-   **ROC-AUC** is still a useful broad, threshold-free ranking and discrimination diagnostic:
    -   ROC-AUC summarizes how well the model ranks fraud above legitimate transactions across all possible decision thresholds.
    -   It is familiar, easy to benchmark, and helpful for comparing broad ranking quality across challenger models.
    -   However, it should not be the primary or sole success metric in highly imbalanced fraud detection.
    -   ROC plots the True Positive Rate (TPR, recall) against the False Positive Rate (FPR) at different decision thresholds.
    -   FPR is calculated as $\frac{\text{False Positives}}{\text{All Actual Negatives}}$. When the legitimate class is very large, even a low FPR can still correspond to an operationally unacceptable number of false alerts.
-   **Precision-Recall (PR) curves** are often better aligned with rare-event fraud operations:
    -   The PR curve plots Precision ($\frac{\text{TP}}{\text{TP} + \text{FP}}$) against Recall ($\frac{\text{TP}}{\text{TP} + \text{FN}}$).
    -   Precision directly measures how many of the predicted frauds are actually fraud, which is critical when investigating alerts.
    -   PR curves do not account for true negatives, making them more sensitive to the model's performance on the minority class.
-   **PR-AUC** (Area Under the PR Curve, also called Average Precision or AP) summarizes the PR curve in a single number.
    -   A high PR-AUC means the model can achieve both high precision and high recall for fraud detection.
    -   The baseline (random classifier) PR-AUC equals the prevalence of the positive class, making it easier to assess model quality.
-   **Operational metrics** still matter for deployment:
    -   Precision at top-$k$ or top-$x\%$
    -   Recall at fixed review capacity
    -   Dollar-weighted capture or savings
    -   False positives per true positive
    -   Calibration diagnostics

PR-AUC and AP are conceptually similar, but AP calculates the area under the PR curve using the observed precision values at each threshold rather than an interpolated curve. Industry practice is not "never use ROC-AUC"; rather, ROC-AUC is a useful secondary diagnostic but is insufficient on its own for highly imbalanced fraud problems. In this lab, we use AP (`average_precision`) as the primary model-selection metric while keeping ROC-AUC as a secondary ranking diagnostic. For details, please refer to the scikit-learn documentation on [Average Precision](https://scikit-learn.org/stable/modules/generated/sklearn.metrics.average_precision_score.html#sklearn.metrics.average_precision_score).

**Process:**

1.  Train multiple estimators on the original, imbalanced dataset.
2.  Select the best model using **average precision** as the primary metric while reviewing **ROC-AUC** as a secondary threshold-free diagnostic.
3.  Check for proper probability calibration. Calibrate the predicted probabilities if needed.
4.  Use the tuning set to select an optimal decision threshold with operational metrics in mind (for example, F1-score, precision at top-$k$, recall at fixed review capacity, false positives per true positive, or dollar-weighted capture).
5.  Evaluate the performance of the calibrated model and chosen threshold on the test set using AP, ROC-AUC, and the operational metrics most relevant to the fraud workflow.

## Cost-Sensitive Learning Approach

**Rationale:**

-   Cost-sensitive learning assigns different costs to classification errors.
-   In fraud detection, failing to detect a fraudulent transaction (a false negative) is typically much more costly than incorrectly flagging a legitimate transaction as fraudulent (a false positive).
-   By incorporating these costs, the model is encouraged to pay more attention to the minority (fraud) class.

**Implementation Detail:**

-   One common way to implement cost-sensitive learning is by using sample weights:
    -   Instances from the minority class are given higher weights.
    -   Instances from the majority class are given lower weights.
    -   The weights are inversely proportional to class frequencies.
-   AutoGluon lets us do this without hand-building a separate weight column:
    -   Set the `sample_weight` argument in `predictor_args` to the special string `"balance_weight"`.
    -   AutoGluon then computes inverse-frequency weights and applies them during training.
-   Sample weights are not needed for the test set evaluation.

**Process:**

1.  Specify sample weights for each observation in the training set by setting `sample_weight` to `"balance_weight"`.
2.  Train multiple estimators on the original, imbalanced dataset.
3.  Select the best model with the highest **average precision** on the test set.
4.  Check for proper probability calibration. Calibrate the predicted probabilities if needed.
5.  Select the optimal decision threshold.
6.  Evaluate the performance of the calibrated model and the optimal threshold on the test set.

## Resampling Approach

**Rationale:**

-   Class-based resampling techniques modify the training data's class distribution to create a less extreme dataset for model fitting.
-   Resampling is performed **only** on the training data.
-   The tuning and test sets remain in their original, imbalanced form to reflect reality (i.e., fraud is rare).
-   In fraud detection, the model still needs plenty of legitimate transactions in view. The job is to learn fraud patterns without hiding the wide range of legitimate behavior that can look suspicious at first glance.

A fully balanced 1:1 sample is often a bad default for fraud detection because it creates a training world the business never sees in production. Fraud is rare, legitimate traffic is abundant, and some legitimate customers will always look odd in the raw data.

**Why 1:1 is usually a poor statistical default**

-   It throws away too much useful non-fraud information. Legitimate transactions contain customer, merchant, device, geographic, seasonal, and amount variation that helps define the real non-fraud population.
-   It can inflate false positives because the sampled data teaches the model that fraud is much more common than it really is unless later calibration, weighting, or prior correction is applied.
-   It damages probability calibration. The rank ordering may still be useful, but raw predicted fraud probabilities from a 1:1 sample are usually not business-usable without correction.
-   More non-fraud controls usually improve precision because they help the model distinguish fraud from legitimate weirdness. The gains are often largest when moving from 1:1 toward ratios such as 5:1 or 10:1.
-   It can make the model learn sampling artifacts if the undersampled non-fraud group no longer looks like the real production population.

**Why 1:1 is usually a poor business default**

-   False positives are expensive: they can create declined legitimate transactions, manual review cost, customer frustration, merchant friction, and lost revenue.
-   The business does not operate in a 50% fraud world. The model must prioritize the riskiest cases inside a large sea of legitimate activity.
-   Fraud teams have limited alert capacity, so the model must rank cases well on a realistic population rather than on an artificially balanced slice.
-   Many legitimate transactions look suspicious. The business wants to separate fraud from legitimate weirdness, not just from a tiny puddle of non-fraud examples.

A practical compromise is to reduce the imbalance without pretending fraud and non-fraud are equally common:

| Non-fraud:Fraud ratio | Why it is useful |
|---|---|
| `5:1` | Useful when fraud cases are limited and model training cost matters, but it is still a relatively aggressive rebalance. |
| `10:1` | Strong general-purpose default for rare-event modeling. |
| `20:1` | Better when legitimate behavior is highly diverse and cheap to include. |
| `1:1` | Usually too artificial for deployment-oriented fraud modeling; keep it as a classroom simplification rather than a default. |

**Common Resampling Schemes:**

-   **Random Undersampling (RUS):** Randomly remove instances from the majority class until the class imbalance is less extreme. In practice, a moderate target such as 5:1, 10:1, or 20:1 is often preferable to forcing a 1:1 sample. This can lead to loss of information from the majority class. However, the scheme substantially reduces the dataset size, which can be beneficial for training speed and memory usage.
-   **Random Oversampling (ROS):** Randomly duplicate instances from the minority class. Here too, a moderate target such as 5:1, 10:1, or 20:1 is often safer than a 1:1 sample. This can lead to overfitting on the minority class examples.
-   **SMOTE (Synthetic Minority Over-sampling Technique):** Creates synthetic samples for the minority class by interpolating between actual minority instances. This can help to create a more diverse set of minority examples than simple oversampling.
-   **Hybrid Methods:** Combine oversampling and undersampling. For example, first apply SMOTE to generate synthetic samples for the minority class, then apply random undersampling to the majority class.

**Process:**

1.  Use RUS, ROS, SMOTE, or hybrid to create a less extreme training set. For RUS and ROS, ratios such as 5:1, 10:1, or 20:1 non-events to events are often more defensible than a fully balanced 1:1 sample.
2.  Train multiple estimators on the resampled training set.
3.  Select the best model with the highest **average precision** on the test set (which is imbalanced).
4.  Check for proper probability calibration. Calibrate the predicted probabilities if needed.
5.  Select the optimal decision threshold.
6.  Evaluate the performance of the calibrated model and the optimal threshold on the test set (which is imbalanced).

### Random Undersampling (RUS)

Randomly remove instances from the majority class to reduce the imbalance in the training set. In this lab, 10:1 is the default example, 5:1 is a more aggressive option when fraud cases are limited and training cost matters, and 20:1 is a more conservative option when legitimate behavior is highly diverse.

Setting the ratio to 1:1 is usually a bad idea for fraud data because it discards too many legitimate transactions, narrows the model's view of normal-but-unusual behavior, and can make the fitted boundary depend on a small and potentially unrepresentative slice of non-fraud cases. That can hurt precision and increase false positives after deployment.

```{python}
#| label: resampling-rus-prep

# Keep 1:1 out of the default path; fraud models usually need more non-fraud controls.
target_non_event_to_event_ratio_rus = 10  # Try 5 for a more aggressive resample or 20 for a more conservative one.
sampling_strategy_rus = 1 / target_non_event_to_event_ratio_rus  # imbalanced-learn expects event:non-event, so 5->0.20, 10->0.10, 20->0.05.

    sampling_strategy=sampling_strategy_rus,
    random_state=2025
)
```

### Hybrid: SMOTE + RUS

First apply SMOTE to generate synthetic minority samples, then apply random undersampling to the majority class.

```{python}
#| label: resampling-hybrid-prep

minority_class_count = y_train.value_counts().min()
if minority_class_count <= 1:
    print("Minority class count is too low for SMOTE. Using original data as fallback.")
    X_train_hybrid, y_train_hybrid = X_train.copy(), y_train.copy()
else:
    k_neighbors_hybrid = min(5, max(1, minority_class_count - 1))
    # Use partial balancing: SMOTE to 0.5 (50% of majority), then RUS to 0.8 (80% majority-to-minority ratio).
    # This avoids the extreme overfitting of full SMOTE and the information loss of full RUS.
    smote_hybrid = SMOTE(random_state=2025, k_neighbors=k_neighbors_hybrid, sampling_strategy=0.5)
    X_train_smote_temp, y_train_smote_temp = smote_hybrid.fit_resample(X_train, y_train)
    rus_hybrid = RandomUnderSampler(random_state=2025, sampling_strategy=0.8)
    X_train_hybrid, y_train_hybrid = rus_hybrid.fit_resample(X_train_smote_temp, y_train_smote_temp)
    print(f"Hybrid training data shape: {X_train_hybrid.shape}")
    print(f"Hybrid target distribution:\n{y_train_hybrid.value_counts(normalize=True)}")
```

### Interpreting the Best Model

We select the best-performing model from the comparison results (by highest Test F1 score) and perform interpretability analysis on it.

```{python}
#| label: interpret-best-model-setup

chosen_model_to_interpret = None
chosen_model_name = None
best_strategy = None
pfi_df = None
pdp_ice_sample_data = None
pdp_features_to_plot = []
categorical_for_pdp = []
shap_sample_data = None
shap_values_summary = None
max_display_features = None
top_shap_features = []

# Dynamically select the best model from comparison_results and prepare the
# artifacts needed for the numbered interpretation outputs below.
if comparison_results:
    summary_df = pd.DataFrame(comparison_results)
    best_row = summary_df.loc[summary_df['Test F1 (Optimized)'].idxmax()]
    best_strategy = best_row['Strategy']
    print(f"Best strategy by F1-score: {best_strategy}")

    model_registry = {}
    for var_name, strategy_key, display_name in [
        ('ag_model_s1', "1. Do Nothing", "Strategy 1 (Do Nothing)"),
        ('ag_model_s2', "2. Class Sensitive Evaluation", "Strategy 2 (Class-Sensitive)"),
        ('ag_model_s3', "3. Cost Sensitive Learning", "Strategy 3 (Cost-Sensitive)"),
        ('ag_model_s4a', "4a. RUS", "Strategy 4a (RUS)"),
        ('ag_model_s4b', "4b. ROS", "Strategy 4b (ROS)"),
        ('ag_model_s4c', "4c. SMOTE", "Strategy 4c (SMOTE)"),
        ('ag_model_s4d', "4d. Hybrid (SMOTE+RUS)", "Strategy 4d (Hybrid)"),
    ]:
        if var_name in locals() and locals()[var_name].is_fitted_:
            model_registry[strategy_key] = (locals()[var_name], display_name)

    if best_strategy in model_registry:
        chosen_model_to_interpret, chosen_model_name = model_registry[best_strategy]
    else:
        print(f"Best model ({best_strategy}) not available. Falling back to Strategy 2.")
        chosen_model_to_interpret = ag_model_s2
        chosen_model_name = "Strategy 2 (Class-Sensitive) - Fallback"

    print(f"Interpreting: {chosen_model_name}")

    X_test_with_label = X_test.copy()
    X_test_with_label['TX_FRAUD'] = y_test
    pfi = chosen_model_to_interpret.predictor.feature_importance(data=X_test_with_label, silent=True)
    pfi_df = pfi.reset_index().rename(columns={'index': 'feature', 0: 'importance'})

    pdp_ice_sample_data = X_train.sample(min(1000, len(X_train)), random_state=2025)
    try:
        pdp_features = chosen_model_to_interpret.predictor.features()
        if len(pdp_features) > 5:
            pdp_features_to_plot = pfi_df['feature'].head(5).tolist()
        else:
            pdp_features_to_plot = pdp_features

        cat_features_metadata = chosen_model_to_interpret.predictor.feature_metadata
```

### Prior Correction for Logistic Regression After Class-Based Resampling

When a logistic regression is fit on a class-resampled version of the training set, the fitted probabilities live on the resampled class-prior scale rather than the original training-set scale. For random undersampling or random oversampling, if the sampling depends only on the class label and does not otherwise change the conditional feature distributions $P(X \mid Y=1)$ and $P(X \mid Y=0)$, then the slope coefficients are preserved and only the intercept changes.

Let

-   $\hat\pi_\text{original}$ = `prior_original`, the fraud rate in the original non-resampled training set.
-   $\hat\pi_\text{sampled}$ = `prior_resampled`, the fraud rate in the class-resampled training set.
-   $\widehat{\alpha}_\text{sampled}$ = `intercept_hat_resampled`, the intercept from a logistic regression fit on the sampled training set.
-   $\widehat{\beta}_\text{sampled}$ = the slope vector from the same sampled-data fit.

Write the sampled-data linear predictor as

$$
\widehat{\eta}_\text{sampled}(x)
=
\widehat{\alpha}_\text{sampled}
+
x^\top \widehat{\beta}_\text{sampled}.
$$

If class-1 observations are retained or duplicated by a class-specific factor $c_1$ and class-0 observations by a factor $c_0$, then class-based sampling implies

$$
\frac{P_\text{sampled}(Y=1 \mid X=x)}{P_\text{sampled}(Y=0 \mid X=x)}
=
\frac{c_1}{c_0}
\cdot
\frac{P_\text{original}(Y=1 \mid X=x)}{P_\text{original}(Y=0 \mid X=x)}.
$$

Therefore the sampled and original log-odds differ by a constant:

$$
\widehat{\eta}_\text{original}(x)
=
\widehat{\eta}_\text{sampled}(x)
-
\log\left(\frac{c_1}{c_0}\right).
$$

The same constant can be written in terms of the two marginal class odds because

$$
\log\left(\frac{c_1}{c_0}\right)
=
\operatorname{logit}(\hat\pi_\text{sampled})
-
\operatorname{logit}(\hat\pi_\text{original}).
$$

Substituting gives the post-fit prior correction:

$$
\widehat{\eta}_\text{original}(x)
=
\widehat{\eta}_\text{sampled}(x)
+
\operatorname{logit}(\hat\pi_\text{original})
-
\operatorname{logit}(\hat\pi_\text{sampled}).
$$

Equivalently, only the intercept changes:

$$
\widehat{\alpha}_\text{original}
=
\widehat{\alpha}_\text{sampled}
+
\operatorname{logit}(\hat\pi_\text{original})
-
\operatorname{logit}(\hat\pi_\text{sampled}),
\qquad
\widehat{\beta}_\text{original}
=
\widehat{\beta}_\text{sampled}.
$$

The corrected fitted probability on the original training-set prior scale is

$$
\widehat{P}_\text{original}(Y=1 \mid X=x)
=
\frac{1}{1+\exp\{-\widehat{\eta}_\text{original}(x)\}}.
$$

For fraud data, $\hat\pi_\text{original}$ is usually much smaller than $\hat\pi_\text{sampled}$. Therefore the correction term is usually negative, lowering the intercept and pulling predicted probabilities back toward the original non-resampled training-set fraud rate.

Do not use `sklearn.linear_model.LogisticRegression` for this demonstration. Its default fit uses L2 regularization and the LBFGS solver, whereas the derivation here is meant to align with an unpenalized binomial GLM fit with `statsmodels` via IRLS.

```{python}
#| label: tbl-logistic-rus-prior-correction-diagnostics
#| tbl-cap: "Diagnostics for logistic RUS prior correction"

%matplotlib inline

def logit(p):
    p = np.clip(p, 1e-12, 1 - 1e-12)
    return np.log(p / (1 - p))

def inverse_logit(z):
    return 1 / (1 + np.exp(-z))

# Example: map a logistic regression trained on the RUS data back to the
# original non-resampled training-set class prior.
# Use statsmodels GLM via the formula interface so the fit follows the
# canonical unpenalized binomial IRLS routine for this intercept shift.
rus_glm_data = X_train_rus.copy()
rus_glm_data[target_label] = np.asarray(y_train_rus)
logistic_formula = f"{target_label} ~ {' + '.join(final_features)}"
logistic_rus = smf.glm(
    formula=logistic_formula,
    data=rus_glm_data,
    family=sm.families.Binomial()
).fit()

prior_original = y_train.mean()
prior_resampled = y_train_rus.mean()
intercept_hat_resampled = logistic_rus.params["Intercept"]

linear_predictor_sampled_tuning = np.asarray(
    logistic_rus.predict(X_tuning, which="linear"),
    dtype=float
)
linear_predictor_sampled = np.asarray(
    logistic_rus.predict(X_test, which="linear"),
    dtype=float
)
raw_sampled_probs_tuning = inverse_logit(linear_predictor_sampled_tuning)
raw_sampled_probs = inverse_logit(linear_predictor_sampled)

prior_log_odds_shift = logit(prior_original) - logit(prior_resampled)
intercept_hat_original_scale = intercept_hat_resampled + prior_log_odds_shift

linear_predictor_original_tuning = linear_predictor_sampled_tuning + prior_log_odds_shift
linear_predictor_original = linear_predictor_sampled + prior_log_odds_shift

# Extract the sampled-data linear predictor directly from statsmodels so the
# code mirrors the derivation: the correction is a constant shift in log-odds.

    linear_predictor_original_tuning - linear_predictor_sampled_tuning
)
constant_shift_range_test = np.ptp(
    linear_predictor_original - linear_predictor_sampled
)
```

# Key Conclusions and Recommendations

At this point the comparison is clean: we held the feature set fixed and changed only how the model responds to severe class imbalance.

**Template For Interpreting Your Results:**

Do not copy the discussion below verbatim into a submission. Use it as a guide for writing up what your own runs actually showed.

-   **Strategy 1 (Do Nothing / Original Imbalanced Data):** Treat this as the control case. If it already ranks fraud reasonably well, the feature set is doing more work than you might expect. If ROC-AUC looks respectable but thresholded performance is weak, that is a reminder that broad ranking quality does not automatically translate into a workable alert rule.

-   **Strategy 2 (Class-Sensitive Metric Selection):** Using PR-AUC (`average_precision`) as the primary metric usually lines up better with how fraud teams work, because only a small high-risk slice of transactions will be reviewed. Keep ROC-AUC in the report, but let AP and threshold metrics carry more weight in the decision.

-   **Strategy 3 (Cost-Sensitive Learning with `balance_weight`):** When you have enough data, reweighting is often the least disruptive way to make the learner take fraud more seriously. It keeps the original sample intact and usually creates fewer side effects than aggressive resampling.

-   **Strategy 4 (Resampling — RUS, ROS, SMOTE, Hybrid):** This family asks a practical question: if you are going to rebalance, how far should you go? For fraud data, 1:1 is often too aggressive. Ratios such as 5:1, 10:1, and 20:1 usually preserve more legitimate behavior while still making the fit easier:

    -   **RUS (Random Undersampling):** RUS lowers training cost by throwing away majority-class rows, but the discarded rows may contain exactly the legitimate variation you need.
    -   **ROS (Random Oversampling):** ROS is easy to implement, but duplicated fraud cases can make the model overreact to a small set of examples.
    -   **SMOTE:** SMOTE creates synthetic minority samples by interpolation. It can add variety relative to ROS, but it still changes the training distribution and usually hurts raw calibration.
    -   **Hybrid (SMOTE + RUS):** Hybrid pipelines can be effective, but they buy flexibility by adding more moving parts to debug and recalibrate.
    -   **Probability Calibration:** Any resampling scheme can distort score calibration. If the probabilities will be used as probabilities rather than just rankings, recalibrate on an untouched tuning set.
    -   **Performance:** Do not assume resampling wins by default. In many fraud problems, better features, a better selection metric, or class weights buy more than a more complex sampling pipeline.

**General Recommendations:**

1.  **Do Not Rely on One Metric:** In imbalanced fraud detection, ROC-AUC is a useful secondary view of ranking quality, not the whole report card. Pair it with PR-AUC (`average_precision`) and operations-aware metrics such as precision at top-$k$/top-$x\%$, recall at fixed review capacity, dollar-weighted capture or savings, false positives per true positive, and calibration diagnostics.

2.  **Try Cost-Sensitive Learning Before Aggressive Resampling:** If you have enough data and want to avoid synthetic examples or heavy undersampling, `sample_weight='balance_weight'` is often the cleanest next step.

3.  **Resample Carefully, Not Automatically:** Do not manipulate training data (e.g., via RUS, ROS, SMOTE) solely because it is imbalanced or feels difficult to model. Such methods can introduce their own problems:

    -   **Use Moderate Targets:** For RUS and ROS, ratios such as 5:1, 10:1, or 20:1 non-events to events are often more realistic than 1:1.
    -   **Preserve Legitimate Variation:** More non-fraud controls help the model learn the sea of legitimate activity rather than a tiny puddle of non-fraud examples.
    -   **Loss of Information (RUS):** Undersampling can discard valuable data from the majority class, especially if you force the data all the way to 1:1.
    -   **Overfitting (ROS, SMOTE):** Oversampling can lead to models that are too specific to the (potentially synthetic) minority samples, and a 1:1 ROS sample makes that risk worse.
    -   **Avoid False-Positive Inflation:** A fraud-heavy sampled training set can teach the model a world where fraud is much more common than it really is, which may translate into more alerts than the business can absorb.
    -   **Destroyed Probability Calibration:** This is a major issue, especially if the model outputs are used for more than just binary classification (e.g., risk scoring, estimating expected fraud rates). More aggressive rebalancing usually makes calibration harder.
    -   **Increased Complexity:** More complex pipelines can be harder to maintain, understand, and debug.

4.  **Calibrate If Scores Will Be Used as Probabilities:** If the model scores feed queue priority, expected-loss estimates, or any downstream decision that depends on the size of the probability, good ranking is not enough. Calibrate on untouched data that reflects the real production population.

5.  **Tune Thresholds on a Separate Set:** Regardless of the imbalance strategy, always use a dedicated tuning set to find an optimal decision threshold that aligns with business objectives (e.g., maximizing F1-score, or balancing precision and recall according to specific cost considerations).

6.  **Spend Time on Features:** In fraud work, a small set of good recency, customer-history, and terminal-history features usually matters more than switching from one imbalance trick to another.
